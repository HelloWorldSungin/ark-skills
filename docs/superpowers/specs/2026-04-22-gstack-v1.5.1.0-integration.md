## Problem Statement

gstack just upgraded from v0.18.3.0 to v1.5.1.0, a major jump. Several new features have direct value for ark-workflow's continuity, observability, and model-routing logic. But ark-workflow shipped its own continuity primitives (`.ark-workflow/current-chain.md`, atomic-lock pattern, `/wiki-handoff` bridges) before v1.5.1.0 existed. The job is to integrate selected gstack v1.5.1.0 features without duplicating state, breaking the atomic chain-file lock invariant, or violating the "Path B is gstack-independent" doctrine from S012.

## What Makes This Cool

Continuous-checkpoint mode makes chain reasoning git-greppable across worktrees and clones. That fixes a long-standing pain: `.ark-workflow/current-chain.md` is gitignored, so chain reasoning evaporates the moment you switch worktrees or clone fresh. After v1.20.0, `git log --grep "WIP:"` rebuilds the reasoning thread.

`/benchmark-models` replaces heuristic `/ccg` substitution decisions with measured cost/quality data. Today we substitute `/ccg` into Heavy chains based on the doctrine "multi-model review pays off here." After v1.21.0, that statement gets evidence behind it.

Together, the wave moves ark-workflow from "we think this is the right routing" to "we know it is, and we can prove it from git history."

## Constraints

Carried forward from prior sessions, NOT to be re-litigated:

- **"Replace, don't stack" review authorities** (S012). Heavy chains substitute `/ccg` with `/autoplan` when gstack is healthy, never stacked.
- **Path B is gstack-independent.** OMC engines (`/autopilot`, `/team`) have internal review phases. Don't layer gstack into Path B.
- **`HAS_GSTACK_*` is a semantic skill-list probe**, not a filesystem check. `~/.gstack/` is advisory only.
- **Three-state degradation UX**: healthy / silent-absent / broken-notice.
- **Chain file SoT**: `.ark-workflow/current-chain.md` is the durable identity. New state mechanisms must coordinate with it, not duplicate it.
- **Atomic chain-file lock**: lock-symlink + tempfile-rename pattern from Arkskill-007. Git operations must coordinate or live outside the lock.
- **Release discipline**: 2 `/ccg` passes per ship (design + pre-push diff), 2-atomic-commits-per-feature shape from S012.

## Premises

All six accepted by user during /office-hours.

- **P1 (architecture).** Continuous-checkpoint commits drop AFTER the atomic check-off helper completes, outside the lock boundary. The lock guards `current-chain.md` mutation, not git state. A failed `git commit` retries safely without corrupting chain state.
- **P2 (UX).** Continuous-checkpoint integration respects whatever the user has set in gstack config. ark-workflow does NOT force the mode. If user is in `explicit`, no WIP commits drop from chains. If `continuous`, Step 6.5 emits a `WIP: chain step N done` commit with `[gstack-context]` body. **Default for users who have never set `checkpoint_mode` is `explicit`** (verified against gstack v1.5.1.0 behavior), so WIP commits are strictly opt-in.
- **P3 (scope).** `/context-save` in the compaction-recovery menu is purely additive (option-(d) alongside a/b/c). Does NOT replace `/wiki-handoff`. Different contracts: `/wiki-handoff` lands in vault with schema validation, `/context-save` is markdown-only in `~/.gstack/`. **Option (d) explicitly opts OUT of the Wiki-handoff invariant** (`SKILL.md:448`). The invariant binds options (a) and (b) only; option (d) is a separate path that skips vault validation by design and proceeds straight to `/compact`.
- **P4 (scope).** `/benchmark-models` calibration is a one-time run producing a vault artifact at `vault/Compiled-Insights/Model-Calibration-2026-04.md`. Substitution rules in chains stay heuristic in v1.20.0. Continuous adaptive substitution is v2.
- **P5 (boundary).** Path B stays gstack-independent. None of these four changes touch Path B chain definitions.
- **P6 (release shape).** Two-wave release per Approach B: Wave 1 (v1.20.0) = cleanup + wiring, Wave 2 (v1.21.0) = calibration + data-driven chain edits.

## Approaches Considered

### Approach A: Linear bundle (S012-conservative)

Single v1.20.0 with all 4 changes. Single PR. Calibration lands as vault artifact without consumer this wave.
Trade-off: bundled rollback risk; calibration shelf-ware until a future wave.

### Approach B: Two-wave (CHOSEN)

Wave 1 (v1.20.0): cleanup + continuous-checkpoint Step 6.5 wiring + `/context-save` menu addition.
Wave 2 (v1.21.0): `/benchmark-models` calibration run + chain-file substitution-rule edits informed by results.
Trade-off: 2 ship cycles, 4 `/ccg` passes total. Calibration becomes actionable.

### Approach C: Dogfood-driven (creative/lateral)

Ship continuous-checkpoint first as own commit, enable on dev branch, build the rest using it.
Trade-off: dogfooded immediately; risk of WIP-squash edge cases leaking polluted history into v1.20.0.

## Recommended Approach

Approach B. Calibration data only matters if it lands somewhere that consumes it. If `/benchmark-models` results sit in the vault as a dead artifact, we spent effort to produce a reference no one will read. Splitting into two waves makes the calibration land alongside the chain-file changes it justifies. That's the difference between "we ran a benchmark" and "the benchmark changed how we route work."

The discipline-vs-speed trade-off favors discipline. 4 `/ccg` passes is the cost of shipping calibration that demonstrably changes routing behavior, not just a benchmark report.

### Commit 2: `feat(ark-workflow): wire continuous-checkpoint mode into Step 6.5 check-off`

**Files:**
- `skills/ark-workflow/SKILL.md` (Step 6.5 documentation + bash block)

**Lock-boundary placement.** The atomic chain-file lock is acquired and released entirely inside the `--format check-off` Python subprocess (see `skills/ark-workflow/scripts/context_probe.py:119-125`, the `with open(lock_path, "w") as lock_fd:` block). The WIP commit therefore shells out from Step 6.5's bash block AFTER the `python3 ... --format check-off` invocation returns, structurally outside the lock. No coordination required.

**gstack-config resolver.** The `gstack-config` binary is NOT on `PATH`. It lives at `~/.claude/skills/gstack/bin/gstack-config`. Step 6.5 must resolve it before invocation:

```bash
GSTACK_CONFIG="${GSTACK_CONFIG:-$HOME/.claude/skills/gstack/bin/gstack-config}"
```

**Step 6.5 bash addition** (literal snippet to add immediately after the existing `--format check-off` invocation).

**Substitution convention:** `{N}` is an LLM-substituted placeholder at chain-execution time (matches the existing pattern in Step 6.5's `--step-index {N}` invocation). The LLM replaces `{N}` with the literal integer step number before emitting the bash to the tool. Inside the snippet, `{N}` is treated as a static integer literal, not a shell variable.

```bash
# Continuous-checkpoint integration (gstack v1.5.1.0+)
# {N} below is LLM-substituted at composition time per Step 6.5 convention.
if [ -x "$GSTACK_CONFIG" ]; then
  CHECKPOINT_MODE=$("$GSTACK_CONFIG" get checkpoint_mode 2>/dev/null)
  if [ "$CHECKPOINT_MODE" = "continuous" ] && git rev-parse --git-dir >/dev/null 2>&1; then
    STEP_LABEL=$(awk -v n={N} '/^- \[x\]/ { c++; if (c == n) { sub(/^- \[x\] [0-9]+\. /, ""); print; exit } }' .ark-workflow/current-chain.md)
    REMAINING_LABEL=$(awk '/^- \[ \]/ { sub(/^- \[ \] [0-9]+\. /, ""); print; exit }' .ark-workflow/current-chain.md)
    [ -z "$REMAINING_LABEL" ] && REMAINING_LABEL="chain complete"
    git commit --allow-empty -m "$(printf 'WIP: chain step {N} done\n\n[gstack-context]\nDecisions: %s\nRemaining: %s\nSkill: /ark-workflow\n[/gstack-context]\n' "$STEP_LABEL" "$REMAINING_LABEL")" >/dev/null 2>&1 \
      || echo "warning: continuous-checkpoint commit skipped (commit failed; check-off still applied)" >&2
  fi
fi
```

**WIP commit body schema (pinned):**

| Field | Source | Example |
|-------|--------|---------|
| `Decisions` | The completed step's label, parsed verbatim from the just-checked `- [x] N. <label>` line in `current-chain.md` | `/ark-context-warmup — load recent + relevant project context` |
| `Remaining` | The label of the next unchecked `- [ ]` line, or the literal string `chain complete` if no unchecked lines remain | `/office-hours — YC forcing questions on gstack-upgrade integration` |
| `Skill` | Always `/ark-workflow` (the parent orchestrator). Steps are checklist items, not skill invocations. | `/ark-workflow` |

**Failure-mode enumeration.** All conditions silently no-op or warn-and-continue. None block check-off.

| Condition | Behavior |
|-----------|----------|
| `gstack-config` binary not found | Skip the WIP commit entirely. No warning (gstack absence is normal). |
| `gstack-config` returns a value other than `continuous` (including `explicit`, empty, or unexpected string) | Skip the WIP commit entirely. No warning. |
| Not in a git repo (`git rev-parse --git-dir` fails) | Skip the WIP commit entirely. No warning. |
| Detached HEAD | Allowed. `git commit` works in detached state; the WIP commit lands on the detached ref. |
| `git commit` fails for any other reason (pre-commit hook, missing user.email, repo-locked) | Emit `warning: ...` to stderr. Check-off remains applied. The LLM should surface the warning inline if the tool output's stderr is non-empty, but MUST NOT halt the chain. |
| Commit succeeds but post-commit hook fails | Treated as success (post-commit hooks don't block the commit). |

**Author identity:** Default git config. No synthetic bot identity.

**Documentation:** Step 6.5 in `SKILL.md` gets a "Continuous Checkpoint Integration" subsection covering the opt-in/opt-out semantics, the literal bash snippet above, the commit-body schema table, and the failure-mode enumeration table.

### Commit 3: `feat(ark-workflow): add /context-save to compaction-recovery menu (option d)`

**Files:**
- `skills/ark-workflow/scripts/context_probe.py` (--format step-boundary branch)
- `skills/ark-workflow/SKILL.md` (Step 6.5 menu documentation)

**Behavior:** When the step-boundary probe emits a compaction-recovery menu, append a 4th option: `(d) /context-save --no-stage` (lightweight markdown-only context save, then `/compact`).

**Why `--no-stage`:** Default `/context-save` commits dirty files. Mid-chain compaction usually means there's unfinished work in the diff. `--no-stage` saves context only without committing the dirty files.

**Differentiation from existing options:**
- `(a)` `/wiki-handoff` then `/compact` — vault-bound bridge with schema validation
- `(b)` `/wiki-handoff` then `/clear` — same, harder reset
- `(c)` Subagent dispatch — no bridge needed, parent context preserved
- `(d)` `/context-save --no-stage` then `/compact` — NEW, lighter exit when vault validation is overkill
